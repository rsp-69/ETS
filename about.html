Technical Architecture Reference · April 2026

When I say Excel and VBA,
here is what I mean.

The system described below is a national-scale, ERP-grade Product Lifecycle Management ecosystem.
Excel was the runtime environment — the same way a server is an environment.
The engineering inside it follows patterns with direct equivalents in Python, R, and enterprise architecture.

Ten pillars. Each one is proof.

# Pillar Python / R equivalent The evidence
01

In-Memory Data Processing

Datasets loaded entirely into RAM as array structures, transformed without touching the worksheet, written back in a single operation.

NumPy ndarray pandas DataFrame

Load → transform in memory → write once. Same paradigm, different syntax.

National SOH/SOO snapshot loaded into arr_2D arrays. Batch transformations across 62 branches — no worksheet reads mid-loop.

Report execution cut from ~4 hours to ~1 hour on the same hardware through algorithmic optimisation alone. The gain was architectural, not infrastructure.

02

Relational Modelling & Single Source of Truth

Independent modules synchronised through shared key-value structures and a central configuration layer — not duplicated logic.

dict / defaultdict pandas merge() SQLAlchemy

Cross-module sync mirrors a shared database session or singleton config object.

Three-tier item mapping: 6-digit item codes → ANACODEs → 8-digit customer IDs. A relational key chain — not flat lookups.

The leading zero logic that broke both the internal TOM project and external Complexica developers — preserved with 100% fidelity.

03

Algorithmic Efficiency & Bottleneck Elimination

Application-layer overhead systematically suppressed — screen updates, calculation events, and worksheet I/O all disabled during execution.

vectorised pandas lazy evaluation

ScreenUpdating off ≈ suppressing render callbacks. The architectural intent is identical.

Weekly SLA suite reduced from ~2 days to ~3 hours through iterative code optimisation.

Real-time ETA calculator in the status bar during batch execution — equivalent to tqdm progress tracking in Python.

04

ETL Pipeline Design

Data extracted from multiple disparate source systems, transformed through layered logic, validated, and loaded to structured outputs and ERP back-ends — automatically.

pandas read_csv() Airflow DAGs paramiko SFTP

The pipeline stages are architecturally identical.

NCW Data Warehouse: ETL pipelines processed 8 wholesalers' weekly data — each in a different format — normalised into one consistent schema.

ERP sync detection: execution held until data had fully synchronised — equivalent to an Airflow sensor or DAG dependency check.

05

State Machine & Lifecycle Governance

Products moved through a defined 6-stage lifecycle with gates, validation rules, and traceable transitions at every stage.

python-transitions Prefect Airflow

Enum-based state machine with task dependencies and gate conditions.

Six-stage closed-loop: Launchpad → Item Maintenance → Logistics Trident → Lost Sales Engine → Health Shield → Performance Tracker.

28-day Accountability Loop: a time-locked gate that automatically surfaced the review tab — blocking stale forecasts from persisting.

06

Event-Driven & Time-Triggered Execution

Core system operations triggered by time events and data conditions — not manual runs.

cron APScheduler AWS EventBridge

Event-driven triggers responding to time and data state.

7:30 AM forensic snapshot triggered daily, freezing national inventory state before ERP data could change.

Thursday Function: ran automatically after daily reports completed — not someone pressing a button.

07

Role-Based Access Control & Audit Logging

Every significant action logged with username, timestamp, and context. Access to master settings permission-gated.

Python logging Django RBAC FastAPI DI

Middleware-layer audit logging and database-level row security.

Digital signature system: username, timestamp, and file signature captured at every key decision point — a forensic chain of custody across all 6 lifecycle stages.

Quantified bypass cost: manual estimates exceeded tool figures in 29 of 37 branches for Ampol P8. Evidence produced from the audit trail.

08

Fault Tolerance & Graceful Degradation

System designed to continue operating under failure conditions — missing files, network drops, ERP lag, corrupted data — without crashing or producing silent errors.

tenacity circuit breaker try/finally

Cached state for degraded-mode operation. Health checks before pipeline execution.

Previous Day Fallback: if current snapshot was missing or corrupted, the system loaded last valid state, flagged degraded mode, and continued processing.

Zero downtime across a nationally distributed operation.

09

Modular Architecture with Defined Service Boundaries

10 independent modules, each with its own logic, inputs, and outputs — sharing configuration and data through defined interfaces, not duplicated logic.

microservices monorepo shared message bus

Python packages with defined public APIs. FastAPI services sharing a common database layer.

Global Configuration Management: one backend sheet controlled national variables across all 10 modules. Update once, propagate everywhere.

Trident tools had explicit dependency order: Tool 1 must complete before Tools 2 and 3 — equivalent to Airflow task dependencies.

10

Parameterised Reporting Engine

256 reports generated from a single underlying dataset by varying customer group, channel, and branch parameters — not 256 separate scripts.

Jinja2 papermill ReportLab

Parameterised SQL queries feeding a report generator. One engine, N outputs.

Lost Sales Master Macro: one engine generated 256 channel-specific reports across 5 national channels and 62 branches — delivered by 9:00 AM daily.

Dynamic email string construction: recipient list compiled at runtime per report — not a static distribution list.

These ten pillars did not require Python or R to exist. They required systems thinking, architectural discipline, and the ability to solve real operational problems at scale.

The tools were incidental. The logic was not.
